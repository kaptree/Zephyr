# 阶段二：用户认证与权限系统

## 目标
实现用户管理、部门管理、JWT双令牌认证及基于角色的权限控制（RBAC），这是整个系统的安全基石。

## 技术要求

### 1. 数据模型（Gorm）
- **users** 表：完整字段映射，含UUID主键、警号、姓名、部门、角色枚举、职级、联系方式、头像、激活状态
- **departments** 表：树形自关联结构，parent_id指向自身，level标识层级
- **roles_permissions** 表：角色-资源-操作-范围四元组权限矩阵

### 2. JWT认证
- **算法**: RS256（RSA非对称加密）
- **双令牌机制**:
  - Access Token：有效期2小时，携带user_id、role、department_id
  - Refresh Token：有效期7天，用于无感刷新
- **令牌吊销**: Redis维护黑名单，登出时加入
- **中间件**: 自动解析Authorization头，注入用户上下文

### 3. 密码安全
- bcrypt加密存储（cost=12）
- 公安标准复杂度校验：8位以上，含大小写+数字+特殊字符
- 登录失败次数限制（5次/15分钟锁定）

### 4. RBAC权限
- 四级角色：super_admin → dept_admin → group_leader → member
- 数据权限范围：global / department / group / self
- 权限检查中间件：基于Casbin策略模式或自研

## API接口

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/v1/auth/login` | 登录，返回双令牌+用户信息+权限 |
| POST | `/api/v1/auth/refresh` | 刷新令牌 |
| POST | `/api/v1/auth/logout` | 登出，吊销令牌 |
| GET | `/api/v1/auth/me` | 获取当前用户含权限矩阵 |
| GET | `/api/v1/departments` | 获取部门树（支持flat参数） |
| GET | `/api/v1/departments/:id` | 部门详情（含成员数、负责人） |
| GET | `/api/v1/users` | 人员列表（分页、多条件筛选） |
| GET | `/api/v1/users/:id` | 人员详情 |
| GET | `/api/v1/users/visible` | 当前用户可见人员列表 |
| POST | `/api/v1/users` | 创建用户 |
| PUT | `/api/v1/users/:id` | 更新用户 |
| DELETE | `/api/v1/users/:id` | 删除用户（软删除） |
