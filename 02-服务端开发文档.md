# 轻燕工作台 —— 服务端开发文档

> 版本：v2.0  
> 定位：面向公司系统的轻量化智能协同办公支撑解决方案 · RESTful API 服务端  
> 核心职责：任务全生命周期管理、专项工作组一键分发、权限控制、实时协同同步、归档追溯、工作成效分析

---

## 一、项目概述

服务端采用 `Go + Gin + GORM` 技术栈，提供高并发、低延迟的 RESTful API 服务，支撑 Web 前端（Vue 3）的多端数据一致性与实时协同。数据库使用 PostgreSQL，利用 JSONB 类型存储协同画布内容。架构采用经典三层分层：`Handler → Service → Repository`。

---

## 二、技术栈与中间件

| 层级        | 技术选型                    | 说明                                    |
| ----------- | --------------------------- | --------------------------------------- |
| 语言 / 框架 | Go 1.22 + Gin               | 高性能 HTTP 框架                        |
| ORM         | GORM v2 + PostgreSQL Driver | 模型映射、AutoMigrate、关联查询         |
| 数据库      | PostgreSQL 15+              | 主数据库，JSONB / 数组 / ILIKE 全文检索 |
| 缓存        | Redis                       | 会话、热点数据                          |
| WebSocket   | gorilla/websocket           | 实时协同通信                            |
| 认证        | JWT RS256                   | Access Token + Refresh Token 双令牌     |
| 权限        | 自研 RBAC                   | 角色 + 数据权限 Scope 隔离              |
| 日志        | Zap + Lumberjack            | 结构化日志 + 文件轮转                   |
| UUID        | google/uuid                 | 全局唯一标识                            |

## 五、完整 API 路由表

### 5.1 全局约定

- **Base Path**：`/api/v1`
- **认证头**：`Authorization: Bearer <token>`
- **响应格式**：`{ code, message, data, timestamp }`
- **分页**：`page`（默认1）、`page_size`（默认20，最大100）

### 5.2 认证接口

| 方法 | 路径            | Handler        | 说明                   |
| ---- | --------------- | -------------- | ---------------------- |
| POST | `/auth/login`   | Login          | 登录，返回 Token       |
| POST | `/auth/refresh` | RefreshToken   | 刷新令牌               |
| POST | `/auth/logout`  | Logout         | 登出（需认证）         |
| GET  | `/auth/me`      | GetCurrentUser | 当前用户信息（需认证） |

## 六、架构与中间件

### 6.1 三层架构

```
┌─────────────────────────────────────────┐
│  Handler  请求解析 → 参数校验 → 调用 Service → JSON 响应   │
├─────────────────────────────────────────┤
│  Service  业务编排 / 事务管理 / 权限判断（本项目部分直接由 Handler 调用 Repo） │
├─────────────────────────────────────────┤
│  Repository  GORM 数据访问 / 查询构建 / 分页 / 事务       │
├─────────────────────────────────────────┤
│  Model  18 个 GORM 模型 / 表结构 / 关联关系定义            │
└─────────────────────────────────────────┘
```

### 6.2 中间件链

**全局中间件**：`RequestID → Recovery → SecurityHeaders → CORS → RequestLogger → RateLimit`

**认证后中间件**：`AuthMiddleware（JWT 验证 + 用户注入上下文）→ OperationLogger（自动记录操作日志）`

**权限中间件**：`RequireRoles(super_admin, dept_admin, group_leader)`

### 6.3 权限模型

- **角色**：`super_admin` / `dept_admin` / `group_leader` / `member`
- **数据权限 Scope**：`global` / `department` / `group` / `self`
- 任务查询时自动按用户角色附加 WHERE 条件（owner_id / creator_id / department_id / group_id）
- `note.Service` 的 `buildScopeQuery` 函数根据角色动态构建查询条件

## 七、核心业务逻辑

### 7.1 一键创建专项工作组（Create WorkGroup）

1. 接收请求：`name / description / template_type / due_time / members[{user_id,role,sub_group_name}] / tags[]`
2. 生成工作组 UUID，插入 `work_groups` 表
3. 遍历 members：
   - 插入 `work_group_members` 记录
   - 生成 sequence 获取年度序列号
   - 创建 Note（`source_type=assigned`、`group_id` 关联工作组、`owner_id` 为成员、`due_time` 继承、`tags` 继承）
   - 创建 `NoteAssignee` 关联（`role_in_note` 标明组长/组员）
4. 返回完整工作组数据（含成员）

### 7.2 工作组专属任务（Group Notes）

- `GET /groups/:id/notes`：调用 `NoteRepository.ListByGroup(groupID, userID, page, pageSize)`
  - 查询 `notes WHERE group_id = ? AND (owner_id = ? OR creator_id = ?)`
  - Preload Tags / Assignees / Group / Attachments / Reminders
- `POST /groups/:id/notes`：创建任务时强制绑定 `group_id`，支持 `tag_ids` 标签关联

### 7.3 工作组搜索（Search Groups）

- 查询 `GET /groups?keyword=&user_id=&date_from=&date_to=&page=&page_size=`
- `keyword` 对 name 和 description 执行 ILIKE 匹配
- `user_id` 筛选"该用户发起 + 该用户作为成员"的工作组
- `date_from / date_to` 时间范围过滤
- Preload Members.User 和 Initiator

### 7.4 序列号生成

- `NoteRepository.GetNextSerialNumber(year)`：原子性 `INSERT...ON CONFLICT UPDATE` 自增
- 格式：`ZP{YYYY}-{序号}`（如 `ZP2026-0001`）

### 7.5 默认报告模板种子

- `main.go` 启动时执行 `FirstOrCreate`，若数据库无默认模板则自动写入
- 首次 `GET /analytics/report-template` 也会自动创建默认模板

## 八、安全规范

- **密码**：bcrypt 加密存储
- **JWT**：RS256 签名，Access Token 2h，Refresh Token 7d
- **接口限流**：基于令牌桶的 RateLimit 中间件
- **SQL 注入**：GORM 参数化查询，禁止字符串拼接
- **操作审计**：OperationLogger 中间件自动记录所有 POST/PUT/DELETE 操作
