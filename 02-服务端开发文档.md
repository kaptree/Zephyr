# 资警数智·轻燕 —— 服务端开发文档

> 版本：v1.0  
> 定位：轻量化情指行一体化支撑解决方案 · RESTful API 服务端  
> 核心职责：便签全生命周期管理、权限控制、实时协同同步、归档追溯、组织人员服务

---

## 一、项目概述

服务端采用 Go + Gin + Gorm 技术栈，提供高并发、低延迟的API服务，支撑桌面端（WinUI 3）、Web前端（Vue 3）、大屏端的多端数据一致性与实时协同。数据库使用 PostgreSQL，利用其 JSONB 类型存储富文本与协同画布内容，利用数组类型存储标签关联。

---

## 二、技术栈与中间件

| 层级 | 技术选型 | 说明 |
|------|---------|------|
| 语言/框架 | Go 1.22 + Gin | 高性能HTTP框架 |
| ORM | Gorm v2 | 模型映射、迁移、关联查询 |
| 数据库 | PostgreSQL 15+ | 主数据库，支持JSONB、数组、全文检索 |
| 缓存 | Redis | 会话、热点数据、WebSocket房间状态 |
| 实时通信 | Socket.io / Gorilla WebSocket | 协同编辑与大屏同步 |
| 认证 | JWT (RS256) | Access Token + Refresh Token 双令牌 |
| 权限 | Casbin / 自研RBAC | 基于角色的数据权限与操作权限 |
| 文件存储 | MinIO / 本地NAS | 附件、导出Word/Excel存储 |
| 任务队列 | Asynq / 自研 | 盯办提醒定时任务、归档清理 |
| 日志 | Zap + Lumberjack | 结构化日志，按级别输出 |

## 三、数据库设计（核心表结构）

### 3.1 用户与权限

**`users`（人员库）**
- `id` UUID PK
- `username` string UNIQUE（警号/系统账号）
- `name` string（姓名）
- `department_id` UUID FK → departments
- `role` enum：`super_admin`, `dept_admin`, `group_leader`, `member`
- `rank` string（职级，用于审批流）
- `phone`, `email`
- `avatar_url`
- `is_active` bool
- `created_at`, `updated_at`

**`departments`（部门库）**
- `id` UUID PK
- `name` string
- `parent_id` UUID FK（自关联，树形结构）
- `level` int（层级：1局级、2支队、3大队...）
- `leader_id` UUID FK → users（部门负责人）

**`roles_permissions`（权限矩阵）**
- `id` UUID PK
- `role` enum
- `resource` string（如 `note`, `user`, `department`）
- `action` string（如 `create`, `read`, `update`, `delete`, `assign`, `remind`）
- `scope` enum：`global`, `department`, `group`, `self`

### 3.2 便签核心

**`notes`（便签主表）**
- `id` UUID PK
- `title` string(200)
- `content` text（富文本HTML/JSON）
- `content_delta` jsonb（TipTap/Slate增量数据，用于协同合并）
- `color_status` enum：`yellow`, `green`, `red`（黄待办/绿完成/红盯办）
- `source_type` enum：`self`, `assigned`, `collaboration`, `emergency`（来源类型）
- `template_type` enum：`default`, `data_analysis`, `special_project`, `emergency_canvas`, `collaborative_writing`
- `creator_id` UUID FK → users（创建人）
- `owner_id` UUID FK → users（当前负责人）
- `assigner_id` UUID FK → users（指派人，上级下发时填写）
- `department_id` UUID FK → departments（所属部门，用于数据权限）
- `group_id` UUID FK → work_groups（专项工作组ID，可为NULL）
- `is_archived` bool DEFAULT false
- `archive_time` timestamp
- `due_time` timestamp（截止时间）
- `completed_at` timestamp
- `remind_count` int DEFAULT 0（盯办次数）
- `last_remind_at` timestamp
- `serial_no` string（文号，自动生成）
- `created_at`, `updated_at`
- `deleted_at` gorm.DeletedAt（软删除）

**`note_tags`（便签-标签关联）**
- `note_id` UUID FK
- `tag_id` UUID FK
- PK(note_id, tag_id)

**`note_assignees`（便签指派/协同人员关联）**
- `note_id` UUID FK
- `user_id` UUID FK
- `role_in_note` enum：`initiator`, `leader`, `member`, `viewer`（在便签中的角色）
- `feedback_content` text（成员反馈内容）
- `feedback_at` timestamp
- `is_read` bool（是否已读）

**`note_attachments`（附件）**
- `id` UUID PK
- `note_id` UUID FK
- `file_name`, `file_path`, `file_size`, `mime_type`

### 3.3 标签与模板

**`tags`（标签库）**
- `id` UUID PK
- `name` string(50)
- `color` string(7)（十六进制色值）
- `category` string（分类：工作类型/优先级/自定义）
- `scope` enum：`system`, `personal`
- `creator_id` UUID FK（personal时必填，system为NULL）
- `sort_order` int

**`templates`（工作模板）**
- `id` UUID PK
- `name` string
- `type` enum
- `fields` jsonb（模板字段定义，如 [{"name":"工作要求","type":"textarea","required":true}]）
- `layout` string（画布分栏数：1/2/4/6/8）
- `is_system` bool

### 3.4 协同与工作组

**`work_groups`（专项工作组）**
- `id` UUID PK
- `name` string
- `note_id` UUID FK（关联的便签）
- `initiator_id` UUID FK
- `status` enum：`active`, `completed`, `dissolved`

**`work_group_members`（工作组成员）**
- `group_id` UUID FK
- `user_id` UUID FK
- `role` enum：`leader`, `member`
- `sub_group_name` string（子分组名，如"技术组"）

**`collaboration_rooms`（协同房间）**
- `id` UUID PK
- `note_id` UUID FK UNIQUE
- `canvas_data` jsonb（画布分栏内容映射，如 {"col_1":{"user_id":"","content":""}}）
- `last_activity_at` timestamp
- `is_active` bool

### 3.5 台账与日志

**`ledger_entries`（工作台账）**
- `id` UUID PK
- `note_id` UUID FK
- `user_id` UUID FK
- `action` enum：`create`, `update`, `complete`, `archive`, `remind`, `assign`, `export`
- `action_detail` jsonb
- `ip_address`, `user_agent`
- `created_at`

**`reminders`（盯办提醒记录）**
- `id` UUID PK
- `note_id` UUID FK
- `reminder_id` UUID FK → users（盯办人）
- `target_id` UUID FK → users（被盯办人）
- `message` text
- `is_acknowledged` bool（被盯办人是否已确认）

## 四、API接口定义（RESTful规范）

### 4.1 全局约定

- **Base Path**：`/api/v1`
- **Content-Type**：`application/json`
- **认证头**：`Authorization: Bearer <access_token>`
- **响应格式**：统一包装
  ```json
  {
    "code": 200,
    "message": "success",
    "data": {},
    "timestamp": 1714291200
  }
  ```
- **分页参数**：`page`（默认1）、`page_size`（默认20，最大100）
- **排序**：`sort_by=created_at&sort_order=desc`

### 4.2 认证接口

| 接口 | 方法 | 路径 | 请求体 | 响应 | 说明 |
|------|------|------|--------|------|------|
| 登录 | POST | `/auth/login` | `{username,password,device_id}` | `{access_token,refresh_token,expires_in,user}` | device_id用于桌面端绑定 |
| 刷新令牌 | POST | `/auth/refresh` | `{refresh_token}` | `{access_token,refresh_token}` | - |
| 登出 | POST | `/auth/logout` | - | `{success}` | 吊销当前设备令牌 |
| 获取当前用户 | GET | `/auth/me` | - | `{data:User}` | 含权限矩阵 `permissions` |

### 4.3 便签管理接口（核心）

#### 4.3.1 获取便签列表
```
GET /notes
```
**查询参数**：
- `status`：enum `active`, `archived`, `all`（默认active）
- `source_type`：过滤来源类型
- `tag_ids`：array，标签ID数组
- `owner_id`：按负责人过滤
- `creator_id`：按创建人过滤
- `department_id`：按部门过滤
- `color_status`：yellow/green/red
- `keyword`：全文检索（标题+内容）
- `date_from`, `date_to`：创建时间范围
- `is_urgent`：bool，是否即将超时（服务端计算）

**权限控制逻辑**：
- 超级管理员：`WHERE 1=1`
- 部门管理员：`WHERE department_id IN (本部门及子部门)`
- 组长：`WHERE group_id IN (我担任组长的组) OR creator_id = 我 OR owner_id = 我`
- 普通用户：`WHERE creator_id = 我 OR owner_id = 我 OR id IN (我作为assignee的note_id)`

**响应数据**：
```json
{
  "data": [
    {
      "id": "uuid",
      "title": "数据分析任务",
      "content": "<p>分析结果...</p>",
      "color_status": "yellow",
      "source_type": "assigned",
      "tags": [{"id":"","name":"协助查询","color":"#3B82F6"}],
      "creator": {"id":"","name":"王队"},
      "owner": {"id":"","name":"张三"},
      "department": {"id":"","name":"技侦支队"},
      "due_time": "2026-04-30T18:00:00Z",
      "is_archived": false,
      "created_at": "",
      "allowed_actions": ["read","update","complete","remind"] // 当前用户对该便签允许的操作
    }
  ],
  "total": 156,
  "page": 1,
  "page_size": 20
}
```

#### 4.3.2 创建便签
```
POST /notes
```
**请求体**：
```json
{
  "title": "必填，200字内",
  "content": "富文本/HTML，10MB内",
  "tags": ["uuid1", "uuid2"],
  "source_type": "self | assigned | collaboration | emergency",
  "template_type": "default",
  "owner_id": "uuid", // 指派他人时必填，默认自己
  "due_time": "2026-04-30T18:00:00Z",
  "assignees": [ // 协同/专项模式必填
    {"user_id": "uuid", "role_in_note": "member"}
  ],
  "group_config": { // 专项分组模式
    "group_name": "专项工作组",
    "sub_groups": [
      {"name": "技术组", "leader_id": "uuid", "member_ids": ["uuid"]}
    ]
  },
  "canvas_config": { // 即时协同模式
    "columns": 4,
    "column_users": ["uuid1", "uuid2", "uuid3", "uuid4"]
  }
}
```

**业务逻辑**：
1. 校验 `source_type` 与当前用户权限：普通用户禁止 `source_type=assigned` 且 `owner_id` 非自己
2. 事务内完成：插入notes表 → 插入note_tags关联 → 插入note_assignees（含创建人自动作为initiator）
3. 若 `source_type=assigned`，生成待办便签推送事件到Redis队列，桌面端长轮询/WebSocket获取
4. 若含 `group_config`，自动创建work_groups与work_group_members记录
5. 若含 `canvas_config`，自动创建collaboration_rooms记录，初始化画布JSONB
6. 自动生成文号 `serial_no`（格式：`资警轻燕〔2026〕XXXX号`），写入ledger_entries台账
7. 返回完整便签对象，含 `websocket_url`（协同模式时）

#### 4.3.3 获取便签详情
```
GET /notes/:id
```
**权限校验**：同列表过滤逻辑，无权限返回404（防遍历）

**响应扩展**：
- 含完整 `assignees` 数组（含反馈内容、反馈时间）
- 含 `group` 工作组信息（如有）
- 含 `room` 协同房间信息（如有）
- 含 `reminders` 盯办历史
- 含 `attachments` 附件列表

#### 4.3.4 更新便签
```
PUT /notes/:id
```
**请求体**：部分更新，仅传变更字段
```json
{
  "title": "...",
  "content": "...",
  "tags": ["uuid"],
  "due_time": "...",
  "color_status": "red"
}
```

**业务逻辑**：
- 校验编辑权限：仅创建人、owner、上级指派人、同部门管理员可编辑
- 更新 `content_delta`（如有），触发协同房间版本号+1
- 记录 `ledger_entries` 变更日志
- 若修改 `owner_id`，视为转派，需重新校验目标人可见性，并推送新owner桌面端

#### 4.3.5 完成便签（归档）
```
POST /notes/:id/complete
```
**请求体**：
```json
{
  "feedback_content": "工作结果总结...",
  "attachments": [{"file_name":"","file_path":""}]
}
```

**业务逻辑**：
1. 校验：仅 `owner_id` 或管理员可操作
2. 事务：
   - 更新 `notes`：`color_status='green'`, `is_archived=true`, `archive_time=NOW()`, `completed_at=NOW()`
   - 更新 `note_assignees`：当前用户反馈内容写入
   - 插入 `ledger_entries`：action=complete
3. 若为协同便签，检查所有assignees是否均已反馈，若否，标记为"部分完成"
4. 推送WebSocket事件 `note:archived` 到所有关联用户，桌面端便签消失
5. 若发起人设置了"自动汇总"，触发异步任务生成Word报告

#### 4.3.6 盯办提醒
```
POST /notes/:id/remind
```
**请求体**：
```json
{
  "target_id": "被盯办人UUID",
  "message": "请尽快完成数据分析，今日18:00前反馈",
  "remind_type": "normal | urgent"
}
```

**业务逻辑**：
1. 校验权限：仅发起人、组长、部门管理员、超管可盯办
2. 校验目标：target_id必须是该便签的owner或assignee
3. 更新 `notes`：`color_status='red'`, `remind_count+=1`, `last_remind_at=NOW()`
4. 插入 `reminders` 记录
5. 推送：
   - WebSocket `note:reminded` 到target_id所有在线设备
   - 若target_id桌面端在线，直接推送桌面强提醒（红色闪烁）
   - 若离线，进入Redis延迟队列，桌面端上线后补偿推送
6. 若 `remind_type=urgent`，同步发送短信/钉钉/企业微信（如集成）

#### 4.3.7 删除便签
```
DELETE /notes/:id?soft=true
```
- 默认软删除（`deleted_at=NOW()`）
- `soft=false` 仅超管可用，物理删除（需二次确认）
- 归档便签可恢复：`POST /notes/:id/restore` → `is_archived=false`, `color_status='yellow'`

## 五、WebSocket实时协同协议

### 5.1 连接管理
- **Endpoint**：`/ws`（Socket.io）或 `/ws/:note_id`（原生WebSocket）
- **认证**：连接时通过Query参数传递 `token`，服务端校验后绑定 `user_id` 到连接上下文
- **房间管理**：以 `note_id` 为Room ID，同一协同便签用户加入同一Room

### 5.2 事件定义

| 事件名 | 方向 | Payload | 说明 |
|--------|------|---------|------|
| `room:join` | C→S | `{note_id}` | 加入协同房间，服务端返回当前画布全量数据 |
| `room:leave` | C→S | `{note_id}` | 离开房间 |
| `canvas:update` | C→S | `{column_id,content,delta,version}` | 客户端发送栏位内容变更，服务端节流100ms广播 |
| `canvas:sync` | S→C | `{column_id,content,updated_by,version}` | 广播给房间内其他成员 |
| `canvas:command` | S→C | `{command_text,from_name,timestamp}` | 领导指令下发，所有客户端顶部显示红色消息条 |
| `note:reminded` | S→C | `{note_id,reminder_name,message,color_status:'red'}` | 盯办推送，桌面端强提醒 |
| `note:archived` | S→C | `{note_id,archived_by}` | 便签归档，桌面端移除便签 |
| `note:assigned` | S→C | `{note_id,title,from_name}` | 新任务下发，桌面端生成黄色闪烁便签 |
| `presence:update` | S→C | `{online_users:[{user_id,name,avatar}]}` | 房间内在线成员列表 |

### 5.3 协同冲突解决
- 画布内容采用 **Operational Transformation** 简化版：服务端保存 `version` 号，客户端发送需带 `last_version`，若不一致则返回全量数据强制同步
- 单栏编辑锁：用户聚焦某栏时发送 `column:lock`，服务端广播该栏"XX正在编辑"，避免大规模冲突

## 六、核心业务逻辑详解

### 6.1 便签创建与权限校验流程
```
1. 解析JWT获取当前用户角色与部门
2. 根据请求体source_type与owner_id判断合法性：
   - 普通用户：source_type只能是self，owner_id必须等于自己
   - 部门管理员：source_type可为assigned，owner_id必须为本部门人员
   - 超管：无限制
3. 校验assignees是否全部在可见范围内（防越权拉人）
4. 开启数据库事务
5. 插入notes表，自动生成serial_no（PostgreSQL序列）
6. 插入关联表（tags/assignees/groups/rooms）
7. 提交事务
8. 异步：写入ledger_entries、推送通知
```

### 6.2 归档与消失机制
- **触发条件**：用户点击完成、管理员强制归档、到达截止时间自动归档（可选配置）
- **数据状态**：`is_archived=true`，数据不移除，仅过滤不展示于活跃列表
- **桌面端同步**：通过WebSocket推送 `note:archived`，桌面客户端从本地SQLite删除该便签记录，UI移除动画
- **追溯查询**：Web端通过 `/notes?status=archived` 查询，支持全文检索与多维过滤

### 6.3 盯办督办机制
- **倒计时引擎**：Asynq定时任务，扫描 `due_time < NOW() + 2h AND is_archived=false` 的便签
- **自动盯办**：超时前2小时自动向owner发送提醒（黄色→红色过渡）
- **人工盯办**：发起人/组长/管理员手动触发，即时生效
- **桌面强警示**：红色便签 + 系统级通知（Windows Toast） + 任务栏图标闪烁

### 6.4 文号与台账自动生成
- **文号规则**：`资警轻燕〔YYYY〕XXXX号`，YYYY为年份，XXXX为4位递增序号，按年度重置
- **生成时机**：便签创建时自动分配，无论是否归档
- **台账内容**：记录每一次创建、编辑、完成、归档、盯办、导出操作，含操作人、IP、UA、变更前后值（JSONB存储）
- **自然生成**：无需人工干预，Web端台账页面按时间轴展示

### 6.5 数据权限隔离（核心安全逻辑）
- **行级安全（RLS）**：PostgreSQL启用RLS，所有notes表查询自动附加 `department_id` 过滤（可选）
- **应用层过滤**：Gorm查询统一注入 `Scope` 函数，根据 `user.role` 自动拼接Where条件
- **字段级脱敏**：普通用户查看他人便签时，敏感附件路径、内部指令隐藏
- **API级鉴权**：Casbin策略规则，如 `p, member, note, read, self` 表示普通用户只能读自己的便签

## 七、安全规范

- **密码**：bcrypt加密，公司标准复杂度（8位以上，含大小写+数字+特殊字符）
- **JWT**：Access Token有效期2小时，Refresh Token 7天，支持吊销列表（Redis）
- **接口限流**：Gin中间件，按IP+用户限流，如登录 5次/分钟，普通API 100次/分钟
- **文件上传**：限制类型（doc/docx/xls/xlsx/pdf/png/jpg），单文件10MB，病毒扫描（ClamAV集成）
- **SQL注入**：Gorm参数化查询，禁止字符串拼接SQL
- **XSS防护**：富文本内容服务端使用DOMPurify策略清洗，仅允许白名单标签
